API Documentation

User Management, Authentication & Subscriptions

OTP & Firebase Authentication | Before Match ID & After Match ID

Introduction

This document outlines the Application Programming Interface (API) endpoints for user authentication, profile management, and subscription handling. It covers both OTP-based authentication and Firebase authentication methods, detailing the request formats, expected responses, and the state changes associated with different user subscription statuses within the application.

Base URL: http://localhost:5000

Chapter 1: User Onboarding and Profile Management (Before Any Subscription)

This chapter covers the initial steps a user takes, including requesting and verifying OTP (One-Time Password) for authentication, and subsequently updating their profile information.

1.1 OTP Request

POST /auth/request-otp

Sends an OTP to the provided phone number. This is the first step in user registration or login.

Request Body Example:

{
  "phone": "+917004653005"
}

Request Body Fields:

Field Type Required Description
phone string Yes The phone number to which the OTP will be sent, including the country code (e.g., "+91").

Successful Response (200 OK):

{
  "success": true,
  "data": null,
  "error": [],
  "message": "OTP sent successfully"
}

Response Fields:

Field Type Description
success boolean Indicates if the OTP request was successful.
data null No specific data is returned for this endpoint on success.
error array An empty array if no errors occurred.
message string A descriptive message indicating the outcome.

Firebase Authentication Flow:
  1. User initiates Google sign-in on the client side
  2. Firebase SDK handles Google OAuth authentication
  3. Client receives Firebase ID token upon successful authentication
  4. Client sends ID token to /auth/firebase-login endpoint
  5. Server verifies the token with Firebase Admin SDK
  6. Server creates new user or logs in existing user
  7. Server returns JWT token for subsequent API calls
  8. User can update profile using the same endpoints as OTP users

6.3 Super Messages

SuperMessages are a premium feature allowing users to send a direct message to another user before a match is formed, consuming a SuperMessage token.

Q1: Should Basic users receive SuperMessages?

Yes they should receive SuperMessages.

Why?

  • Creates FOMO (fear of missing out): Someone messaged me, I cant reply unless I match or upgrade.
  • Drives premium conversions: unlock reply by upgrading.
  • Increases engagement for both sides.
Business Rule: Basic users can view SuperMessages, but cannot reply until a match is made or they upgrade to a plan that allows replies.

Q2: What if user B wants to reply but not match?

Don't allow that replying = engaging = match should be formed.

Best Business Logic:

  • Replying to a SuperMessage = Accepting the match.
  • Once User B replies create a match (if not already).
  • Move all further chat to the MatchChat system.
  • SuperMessage becomes the first message.
UX Flow (SuperMessage Reply):

SuperMessage arrives: "Abhishek sent you a message: 'Hey there '"

Options shown:

  • "Reply and match" proceeds to MatchChat
  • "Ignore" no action
